# Commit Convention

A standardised commit convention for projects using the mzspec delivery pipeline.
Based on [Conventional Commits](https://www.conventionalcommits.org/) with OpenSpec extensions.

## Format

```
<type>(<scope>): <description>

[optional body]

[optional footer(s)]
```

The **type** and **description** are required. The **scope** is recommended (especially for
OpenSpec changes). The subject line must be **≤ 72 characters**.

## Types

| Type       | When to use                                                  | Example                                              |
|------------|--------------------------------------------------------------|------------------------------------------------------|
| `feat`     | A new feature or enhancement                                 | `feat: add LLM-backed profile extraction`            |
| `fix`      | A bug fix                                                    | `fix: handle empty session in extraction`            |
| `spec`     | Spec contract changes (OpenSpec proposal, delta, sync)       | `spec(c0005): define profile extraction scenarios`   |
| `docs`     | Documentation only (README, guides, ADRs, inline comments)   | `docs: add local development setup guide`            |
| `chore`    | Maintenance, tooling, CI, config, dependencies               | `chore: update pyproject.toml dependencies`          |
| `refactor` | Code change that neither fixes a bug nor adds a feature      | `refactor: extract scrubber into shared module`      |
| `test`     | Adding or updating tests (no production code change)         | `test: add regression test for PII scrub ordering`   |
| `style`    | Formatting, linting, whitespace (no logic change)            | `style: sort imports per ruff`                       |
| `perf`     | Performance improvement                                      | `perf: cache term-frequency map across runs`         |
| `revert`   | Reverting a previous commit                                  | `revert: restore removed fallback path`              |
| `build`    | Build system, toolchain, packaging                           | `build: bump uv lock after dependency add`           |
| `ci`       | CI/CD configuration (GitHub Actions, gates)                  | `ci: add nightly LLM-provider gate`                  |

## Scope

The scope identifies the **module** or **OpenSpec change** the commit belongs to.

### OpenSpec change scope (preferred when working inside a change)

Use the change slug (without the `cNNNN-` prefix for brevity, or the full slug):

```
feat(llm-profiling): store term-frequency map and topic weights
fix(c0005-llm-profiling): escape untrusted strings in report
spec(c0005): define extraction scenarios
```

### Package scope (for non-OpenSpec changes or cross-cutting fixes)

Use the package or module name:

```
feat(persona): add roleplay session generator
fix(memic_store): handle missing version field on read
docs(behavior): add docstrings to all extractors
```

### No scope

For simple, cross-cutting, or obvious changes:

```
chore: update .gitignore
docs: fix typo in README
```

## OpenSpec commit conventions

When working on an OpenSpec change (`feat/<change>` branch), follow these patterns:

### Feature commits (during implementation)

```
feat(<change>): <description>

feat(c0005): store term-frequency map and topic weights
feat(c0005): add LLM extraction path with PII-scrub ordering
feat(c0005): render word cloud and topic chart in report
```

Label the unit from `tasks.md` at the end in parentheses when it completes one:

```
feat(c0005): store term-frequency map (unit 01)
feat(c0005): add LLM extraction path (unit 02)
```

### Fix commits (during review/address-review)

Use the change slug and reference the review finding:

```
fix(c0005): escape untrusted strings in report (review finding #4)
fix(c0005): parse reasoning-model responses (review finding #5)
```

### Spec commits (spec PR — the spec-only PR before code)

```
spec(c0005): define profile extraction scenarios
spec(c0005): add word cloud and topic chart requirements
```

Spec commits are merged via the **spec PR** (human merges). They become the contract on `main`.

### Chore commits (evidence, changelog, end-of-change)

```
chore(c0005): evidence, changelog
chore(c0005): add test results and coverage report
```

## Commit body

Use the body to explain **why** the change was made, not **what** (the diff shows what):

```
feat(c0005): add PII scrub at extraction boundary

Scrub PII once at run_extraction and use the scrubbed copy for
both term-frequency map and LLM corpus. This enforces the
no-raw-PII guarantee at the function boundary rather than relying
on callers to remember.
```

## Commit trailer (mzspec projects)

All commits generated by the pipeline include:

```
Co-Authored-By: Claude <noreply@anthropic.com>
```

This is appended automatically by `ship-code`, `address-review`, and related workflows.
Human-authored commits should NOT include this trailer.

## Squash merge convention (merge-pr workflow)

When using `merge-pr` with `--strategy squash`, the squash commit title is auto-derived
from the PR title. **Best practice**: ensure the PR title follows the commit convention:

```
✅ feat(c0005): add LLM-backed profile extraction with word cloud and topic chart
❌ Feature implementation
❌ Add new stuff
```

The `merge-pr` workflow accepts `--title` to override the squash commit title explicitly:

```bash
/opsx:merge-pr c0005-llm-profiling-inspector --title "feat(c0005): add LLM-backed profile extraction"
```

## Changelog integration

CHANGELOG entries are grouped by type, matching the commit types:

```
### Added
- feat: LLM-backed profile extraction...
- feat: Profile report inspector...

### Changed
- refactor: Report profile reads break version ties...

### Fixed
- fix: Escape untrusted strings in report...

### Removed
- ...
```

See [Keep a Changelog](https://keepachangelog.com/) for the full format.

## Quick reference

```
# Feature (OpenSpec change)
feat(c0005): store term-frequency map and topic weights

# Feature (package scope)
feat(persona): add roleplay session generator

# Bug fix
fix(c0005): handle empty session in extraction

# Spec contract
spec(c0005): define extraction scenarios

# Documentation
docs: add local development setup guide

# Maintenance
chore: update uv.lock

# Refactoring
refactor(persona): extract scrubber into shared module

# Tests
test(c0005): add regression test for PII scrub ordering
```
